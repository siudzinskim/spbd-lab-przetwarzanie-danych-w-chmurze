Lab 01: Eksploracja i DuckDB

Cel: Generowanie danych i pierwsze kroki z silnikiem analitycznym.

Dane wejściowe

  • Kaggle: Books Sales and Ratings (CSV).
  • Dane syntetyczne: Generowane skryptem Python (Klienci, Transakcje).
  • Narzędzia: DuckDB CLI.

Instalacja DuckDB


# Instalacja CLI
curl https://install.duckdb.org | sh

# Dodanie do ścieżki (bashrc)
echo 'export PATH=$PATH:/config/.duckdb/cli/latest/' >> ~/.bashrc
source ~/.bashrc

Praca z bazą danych


-- Podłączenie do bazy plikowej
duckdb bookstore.ddb

-- Skrócona składnia DuckDB
FROM books;

-- Import danych z zewnętrznych plików
.import customers.csv customers

CREATE OR REPLACE TABLE transactions AS
SELECT * FROM read_json_auto('transactions.json');

Wnioski z Lab 01

  • DuckDB potrafi czytać dane "w locie" (widoki zewnętrzne).
  • Automatyczne wykrywanie schematów dla JSON (`read_json_auto`).
  • Wysoka wydajność przy minimalnej konfiguracji.

Lab 02: Transformacja z dbt

Cel: Budowa pipeline'u transformacji w standardzie Analytics Engineering.

Inicjalizacja dbt


dbt init dbt_project
# Adapter: duckdb

Struktura projektu:

  • models/ - tu piszemy SQL
  • seeds/ - statyczne CSV
  • analyses/ - zapytania ad-hoc

Konfiguracja (profiles.yml)


dbt_project:
  outputs:
    dev:
      type: duckdb
      path: /ścieżka/do/bookstore_dwh.ddb
      threads: 2
  target: dev

Weryfikacja: dbt debug

Seeds & Sources

Seeds: Pliki CSV (np. klienci) ładowane poleceniem dbt seed.

Sources: Deklaracja danych źródłowych w schema.yml.


sources:
  - name: raw_data
    tables:
      - name: books

Architektura Modeli

Staging

Oczyszczanie, rzutowanie typów, zmiana nazw.

Mart/Fact

Łączenie tabel (Joins) i agregacja biznesowa.

Przykładowy Model Fact


{{ config(materialized='table') }}

with transactions as (
    select *, unnest(items) as item from {{ ref('stg_transactions') }}
),
customers as ( select * from {{ ref('stg_customers') }} )

select
    t.transaction_id,
    c.email,
    (t.item.unit_price * t.item.quantity) as total_amount
from transactions t
left join customers c on t.customer_id = c.customer_id

Testy i Jakość Danych

Definiowane w plikach YAML:


columns:
  - name: customer_id
    tests:
      - unique
      - not_null
      - relationships:
          to: ref('stg_customers')
          field: customer_id

Komenda: dbt test

Dokumentacja (DAG)

Automatyczne generowanie dokumentacji projektu.


dbt docs generate

Podsumowanie

  • DuckDB: Idealny do analiz lokalnych i chmurowych bez serwera.
  • dbt: Standard w inżynierii danych – wersjonowanie, testy i czytelność.
  • Efekt: Skalowalna i łatwa w utrzymaniu hurtownia danych.
