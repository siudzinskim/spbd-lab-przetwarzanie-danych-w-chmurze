dbt-core

Silnik transformacji danych

1.1 Filozofia dbt

dbt (Data Build Tool) to warstwa transformacji, która pozwala traktować kod SQL jak kod aplikacji.

  • Wersjonowanie: Git jako źródło prawdy.
  • Testowanie: Walidacja danych przed ich udostępnieniem.
  • CI/CD: Automatyczne wdrażanie zmian.

1.2 ELT vs ETL

W tradycyjnym ETL transformacja odbywa się poza bazą. W dbt (ELT) transformacja dzieje się wewnątrz hurtowni danych.

Push-down computation: dbt wysyła kod SQL, baza wykonuje pracę.

1.3 Rola Analytics Engineer

Osoba, która stosuje praktyki Software Engineering (DRY, testy, dokumentacja) do modelowania danych.

1.2 Od SQL do Engineeringu

dbt wprowadza do świata baz danych koncepcje znane z inżynierii oprogramowania:

  • Modułowość (kod wielokrotnego użytku).
  • Automatyczne testowanie.
  • Kontrola wersji (Git).
  • Ciągła integracja (CI/CD).

1.3 Paradygmat ELT

dbt nie jest narzędziem typu ETL. Działa w modelu ELT.

Dane są najpierw ładowane do bazy (Extract & Load), a dbt wykonuje ich transformację (Transform) bezpośrednio w silniku bazy danych.

Gdzie dbt znajduje się w Modern Data Stack?

2.1 Miejsce dbt w rurociągu

Grafika: Oficjalna dokumentacja dbt Cloud Overview

2.2 dbt jako "Integration Tool"

dbt stanowi warstwę spajającą surowe dane z narzędziami BI (Tableau, Looker, PowerBI).

Integracja: dbt pobiera metadane z różnych źródeł, transformuje je i wystawia gotowe tabele "biznesowe".

dbt-core kompiluje pliki .sql oraz .yml do natywnego kodu SQL specyficznego dla Twojej bazy (np. DuckDB, Snowflake).

Wynik kompilacji znajduje się w folderze target/.

2.4 Directed Acyclic Graph (DAG)

Każdy projekt dbt to graf zależności. dbt wie, w jakiej kolejności uruchomić modele na podstawie funkcji ref().

2. Inicjalizacja i Profiles

2.1 Anatomia dbt init

Komenda dbt init buduje strukturę katalogów. Kluczowe pliki:

  • dbt_project.yml: Serce projektu.
  • profiles.yml: Dane dostępowe do bazy (trzymane poza Git).

2.2 dbt_project.yml w detalu


name: 'my_lab_project'
version: '1.0.0'
config-version: 2
profile: 'default' # Mapowanie do profiles.yml

models:
  my_project:
    staging:
      +materialized: view # Domyślnie widoki
    marts:
      +materialized: table # Domyślnie tabele

2.3 Zarządzanie Środowiskami (Targets)

W profiles.yml definiujemy cele (dev, prod, staging). Pozwala to na separację danych testowych od produkcyjnych.

2.4 Walidacja Połączenia

dbt debug – komenda sprawdzająca profil, połączenie z bazą i wersję adaptera.

3. Dane Wejściowe: Seeds i Sources

3.1 Czym są Seeds?

Pliki CSV w folderze seeds/. Przeznaczone dla danych statycznych, które nie zmieniają się często.

Uwaga: dbt nie jest narzędziem do ładowania Gigabajtów danych przez Seeds!

3.2 Automatyczne typowanie i błąd typowania

dbt wspiera automatyczne wykrywanie typów (sniffing), ale w pewnych okolicznościach zawodzi przy specyficznych formatach.

3.3 Rozwiązanie: Wymuszanie Typów (YAML)


seeds:
  - name: customers
    config:
      column_types:
        customer_id: bigint
        registration_date: date
        postal_code: varchar

3.4 Definiowanie Źródeł (Sources)

Sources pozwalają na nazwanie "surowych" tabel w bazie. Pozwala to na śledzenie świeżości danych (Freshness).

3.5 Funkcja source() w kodzie


-- models/staging/stg_books.sql
SELECT * FROM {{ source('raw_data', 'books') }}

4. Modele i Materializacje

4.1 Serce dbt: SELECT

Model to po prostu plik SQL. Nie piszesz DDL. dbt zawija Twój SELECT w CREATE TABLE/VIEW AS ....

4.2 Funkcja ref() i DAG

Najważniejsza funkcja dbt. Buduje zależności między modelami.


SELECT * FROM {{ ref('stg_customers') }}

4.3 DAG (Lineage)

Dzięki ref() dbt wie, że model B musi zostać zbudowany po modelu A. Zapobiega to błędom brakujących tabel.

4.4 Materializacja: View

Domyślna. Szybka w budowie, wolna przy odczycie (SQL uruchamiany za każdym razem).

4.5 Materializacja: Table

Fizyczny zapis danych. Wydajny przy odczycie, zajmuje miejsce na dysku.

4.6 Materializacja: Incremental (Deep Dive)

Pozwala dołączać tylko nowe rekordy. Kluczowe dla dużych zbiorów (BigData).


{{ config(materialized='incremental', unique_key='id') }}
SELECT * FROM ...
{% if is_incremental() %}
  WHERE updated_at > (select max(updated_at) from {{ this }})
{% endif %}

4.7 Materializacja: Ephemeral

Modele efemeryczne nie istnieją w bazie. Są wstrzykiwane jako CTE do modeli zależnych.

5. Potęga Jinja i Makr

5.1 Czym jest Jinja?

Język szablonów, który dodaje do SQL pętle, warunki i zmienne.

5.2 Zmienne (Variables)

Definiowane w dbt_project.yml lub przekazywane z CLI (--vars).

5.3 Przykład pętli FOR


{% set payment_methods = ["bank_transfer", "credit_card", "gift_card"] %}

select
    order_id,
    {% for method in payment_methods %}
    sum(case when method = '{{method}}' then amount else 0 end) as {{method}}_amount
    {{ "," if not loop.last }}
    {% endfor %}
from {{ ref('raw_payments') }}
group by 1

5.4 Własne Makra

Reużywalne funkcje SQL trzymane w folderze macros/.

5.5 Pakiety (dbt Packages)

Zewnętrzne biblioteki (np. dbt-utils). Instalowane przez dbt deps.

6. Testowanie i Jakość Danych

6.1 Testy Generyczne

Implementowane w YAML. Cztery standardowe testy dbt:

  • unique
  • not_null
  • accepted_values
  • relationships

6.2 Przykład Testu Relationships


columns:
  - name: customer_id
    tests:
      - relationships:
          to: ref('stg_customers')
          field: customer_id

6.3 Testy Osobliwe (Singular)

Pliki SQL w folderze tests/. Jeśli zapytanie zwróci jakikolwiek wiersz – test kończy się porażką.

6.4 Severity: Warn vs Error

Konfiguracja pozwalająca zdecydować, czy błąd testu ma zatrzymać pipeline, czy tylko ostrzec.
