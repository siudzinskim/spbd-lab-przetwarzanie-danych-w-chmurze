Problem: Spaghetti SQL

W Lab 01 ręcznie importowaliśmy dane. W Lab 02 wprowadzamy porządek:

  • Wersjonowanie: Kod SQL w plikach, nie w konsoli.
  • Zależności: dbt wie, co uruchomić najpierw.
  • Modularność: Reużywalne fragmenty kodu.

Stack Technologiczny

Transformacja

dbt (Data Build Tool)

Silnik Bazy

DuckDB

Formaty Danych

CSV & JSON

DuckDB służy jako lokalna "mini-hurtownia", idealna do prototypowania.

Seeds

Ładowanie statycznych plików CSV (np. słowniki, klienci) prosto do bazy.


							dbt seed

Sources

Deklarowanie danych "surowych", które już są w bazie, a którymi dbt nie zarządza bezpośrednio.

Models

Pliki .sql zawierające kwerendy SELECT. dbt zamienia je w tabele lub widoki.

Budowa Grafu (DAG)

Zamiast wpisywać nazwy tabel na sztywno, używamy funkcji Jinja:


						-- Odwołanie do źródła
						FROM {{ source('raw_data', 'books') }}

						-- Odwołanie do innego modelu (zależność!)
						FROM {{ ref('stg_customers') }}

To pozwala dbt zbudować Directed Acyclic Graph i zarządzać kolejnością uruchamiania.

Co zrobimy?

  1. Inicjalizacja projektu: dbt init.
  2. Konfiguracja profiles.yml dla DuckDB.
  3. Warstwa Staging: Ujednolicenie nazw i typów danych.
  4. Obsługa JSON: Czytanie transakcji bezpośrednio w dbt.
  5. Warstwa Fact: Budowa tabeli faktów (Join klientów, książek i transakcji).
  6. Materializacja: Decydowanie, co jest widokiem, a co tabelą.

Data Quality

Lab 02 uczy, że kod to nie wszystko:

  • Tests: Automatyczna weryfikacja unique i not_null.
  • Docs: Generowanie interaktywnej dokumentacji projektu i grafu zależności.

						dbt test
						dbt docs generate && dbt docs serve
